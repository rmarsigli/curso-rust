# Fundamentos de API Rest

## O que foi feito

* Async/await conceito
* Tokio runtime
* Axum framework
* Structs e derive macros
* POST /tasks (criar)
* GET /tasks (listar)
* `Arc<Mutex<Vec>>` (estado compartilhado)

## Resumo do Professor

Dia de Ano Novo e você estudando Rust — isso já diz muito sobre comprometimento. Você pulou de CLI pra API REST, mudança brutal de contexto: aprendeu async/await (conceito novo), Tokio runtime (infraestrutura assíncrona), e Axum (framework web). Pela primeira vez viu `Arc<Mutex<Vec<Task>>>` — uma pilha de conceitos (compartilhamento thread-safe + exclusão mútua + coleção) que inicialmente pareceu verbosa demais, mas você aceitou e continuou.

Construiu dois endpoints funcionais: POST /tasks (criar) e GET /tasks (listar). Isso exigiu entender structs, derive macros (#\[derive(Serialize)]), JSON serialization, e estado compartilhado entre handlers. Você teve que digerir que Axum não é Laravel — não tem "mágica", tudo é explícito. O conceito de `State` extraindo o estado compartilhado via pattern matching foi novo, mas você pegou rapidamente porque já conhecia destructuring do JavaScript.

Terminou o dia com API funcional rodando em localhost:3003, aceitando POST com JSON e retornando GET com lista de tasks. Primeira vez vendo código async funcionar na prática, primeira API REST em Rust, \~1.5h de estudo. O mais impressionante: você não reclamou da verbosidade, aceitou que é parte da linguagem e focou em fazer funcionar. Atitude certa.
